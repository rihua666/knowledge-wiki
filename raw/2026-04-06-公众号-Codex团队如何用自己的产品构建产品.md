# 访谈｜Codex 团队如何用自己的产品构建产品——整个 Spec 只有 10 个要点

> 抓取日期: 2026-04-13
> 标签: #openai #codex #产品管理 #agent #开发工具

---

OpenAI Codex 团队的产品规格文档只有 10 个要点。不是说每个功能的文档只有 10 个要点，而是整个产品的 spec 就这么多。设计师写的代码量超过了六个月前工程师写的。50 到 100 人的团队，直到最近才有了第二个产品经理。

这些数字来自 OpenAI Codex 产品负责人 Alexander Embiricos 和开发者体验负责人 Romain Huet。两人在 Peter Yang 主持的 Behind the Craft 节目中，详细讲述了 Codex 团队如何用自己的产品构建产品，为什么中期路线图在 OpenAI 行不通，以及 PM 这个岗位可能正在从"领导者"变成"填空人"。

原始视频：https://www.youtube.com/watch?v=9qXc-THAvc0

要点速览：
- Codex 团队几乎不写产品规格文档，只有涉及多人协调或重大决策时才写，而且往往只有 10 个要点的长度
- OpenAI 内部的规划哲学是只做近期（8 周以内）或远期（方向感），永远不做中期产品路线图
- GPT-5.2 Codex 在 2025 年 12 月发布是关键转折点，模型能力跨过了"可以可靠长时间独立工作"的门槛
- Codex 团队刻意保持"海盗船"式运作，50-100 人的团队长期只有一个 PM
- Alex 认为 PM 不是领导岗位，而是"填补空缺"的岗位

Peter Yang 开门见山：你们还写 spec 吗？

Alex 回答：Codex 团队几乎不写产品规格文档。只有在问题复杂到一个人脑子装不下、需要多人协调的时候才会写。而且即使写了，也就十来个要点。"让离金属最近的人做尽可能多的决策"是核心原则。

Romain 补充了 Codex 的 plan mode（规划模式）：给 Codex 一个项目，不给任何具体指令，只说"我们下一步该做什么？"Codex 会自动分析代码库现状，提出几个方向让人类选择。产品规划不再是人类先想清楚再写文档，而是人和 AI Agent 一起在代码层面探索可能性。

Alex 作为 PM 使用 Codex 的三种模式：
1. 简单改动直接上手，用 Codex 生成代码、测试、提交 PR
2. 中等复杂度的改动，让 Codex 先做一个实现计划
3. 模糊想法：直接在 Codex 里和模型对话，让它去代码库里探索、提问、生成方案。通过这个过程建立对问题的理解，然后把这份理解分享给工程师。"我不是在写代码，我是在建立心智模型。"

设计师角色：Codex 团队的设计师现在写的代码量，超过了六个月前一个工程师写的代码量。

规划方法：在 OpenAI，你要么做近期规划，要么做远期规划，但永远不要做中期规划。近期是 8 周以内，远期是一种"感觉"。中间那段？基本不存在。

Codex App 的诞生：2025 年 12 月 GPT-5.2 Codex 发布后，用户开始用 tmux 同时运行大量 Codex 会话。社交媒体上出现 Peter Steinberger 在三块显示器上同时开了约 18 个终端窗口运行 Codex 的照片。但只有顶尖 1% 的工程师会用终端并行的方式工作，如何让这件事变得直觉化？最后推动项目启动的不是一份 spec，而是一份说明"为什么我们认为做一个 app 是好主意"的文档。Codex App、IDE 扩展和 CLI 底层共用同一套开源的 Rust 核心（Codex harness）。

核心设计原则：先为 power user 做可配置性，再为普通用户做简化。核心交互要做到极简，"让产品几乎隐形，让模型的能力自然显现"。

50-100 人团队的"海盗船"式运作，跨职能协调极少。Alex 有执行模式和协调模式两种工作状态。

PM 不是领导岗位，是填空岗位。AI 工具让每个人都能做一部分别人的工作，Scott Belsky 的"人才栈压缩"正在发生。Stripe 在没有任何 AI 工具的情况下做到了 250 名员工零 PM。

招聘标准：能动性（agency）。看作品不看简历，看能动性不看资历。"给我看你做了什么。"

Q&A：
- Codex 团队用什么方式代替产品规格文档？让工程师直接在代码层面做决策
- OpenAI 内部怎么做产品规划？8 周以内的具体冲刺目标 + 一年以上的方向性判断
- PM 还有存在的必要吗？取决于你的团队是否是自己产品的用户
- Codex 团队怎么招人？看作品不看简历，看能动性不看资历
