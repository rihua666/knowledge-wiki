# SCHEMA.md — 个人知识库 Wiki 配置

> 本文件是 Wiki 的 Schema 层，定义结构、约定和工作流。
> LLM 在执行 Ingest/Query/Lint 操作时必须遵循本文件。

---

## 🏷️ 页面类型

| 类型 | frontmatter | 说明 | 示例 |
|------|------------|------|------|
| **汇总页** | `type: hub` | 主题入口，汇总该主题的关键洞察，引用所有细节页 | `上下文工程.md` |
| **细节页** | `type: detail` 或不填 | 单个源文档的深度分析 | `有效上下文工程.md` |

每个分类目录下最多一个 `hub` 页面，是该分类的推荐入口。

---

## 📂 目录结构

```
wiki/
├── index.md          # 全局索引（每页一行摘要）
├── log.md            # 追加式操作日志
├── AI-Agent/         # AI 代理系统架构、方法论、设计模式
├── LLM/              # 大语言模型相关研究
├── 记忆系统/          # 代理记忆机制
├── 评估方法/          # LLM 代理评估
├── 上下文工程/        # 上下文管理和工程实践
└── 项目实践/          # 开源项目和工具
```

## 📄 页面格式

每个 Wiki 页面遵循以下格式：

```markdown
---
title: [标题]
type: hub        # 汇总页（可选），细节页不填或 type: detail
category: [分类名]
sources: [raw/ 中对应文件名, 用逗号分隔]
tags: [标签1, 标签2]
created: [YYYY-MM-DD]
updated: [YYYY-MM-DD]
confidence: high | medium | low | mixed
---

> **type: hub** = 主题汇总页（该主题的入口）
> **无 type / type: detail** = 细节文章（具体源文档的分析）

# [标题]

> [一句话摘要]

## 核心内容

[正文 — 带行内置信度标注]

## 关键要点

- [要点1]
- [要点2]

## 与其他主题的关系

- [[相关页面1]] — [关系说明]
- [[相关页面2]] — [关系说明]

## 来源

- [source1](../raw/文件名) — [简述]
```

## 🔄 三个核心操作

### 1. Ingest（摄入）

**触发**：用户将文件放入 `raw/` 后说"编译知识库"

**流程**：
1. 读取 `raw/` 中的新文件（PDF 先用 `pdftotext` 转换）
2. 提取关键信息：核心论点、方法、发现、数据
3. 与用户讨论要点（可选，用户可选择跳过）
4. 在对应分类目录下创建 Wiki 页面
5. 更新 `index.md` 中对应分类的条目
6. 检查是否需要更新已有的相关页面（交叉引用）
7. 在 `log.md` 追加记录

**注意**：
- 每次只处理一个文件，除非用户要求批量处理
- 单个素材可能需要更新 5-15 个 Wiki 页面
- raw/ 文件永远不修改，只读

### 2. Query（查询）

**触发**：用户针对知识库提问

**流程**：
1. 先读 `index.md` 定位相关页面
2. 深入阅读相关页面
3. 综合信息，生成带引用的回答
4. 如果答案有价值，建议用户归档为新 Wiki 页面

**输出格式**：
- 简单问题：直接回答
- 复杂分析：生成 Markdown 页面 → 归档到 Wiki
- 对比分析：生成表格 → 归档
- 可选：Marp 幻灯片、matplotlib 图表

### 3. Lint（健康检查）

**触发**：用户说"检查知识库"或定期运行

**检查项**：
- [ ] 矛盾检测：不同页面间的冲突信息
- [ ] 过时内容：被新源超越的旧观点
- [ ] 孤立页面：没有入链的页面
- [ ] 缺失页面：被引用但未创建的概念
- [ ] 交叉引用完整性：相关页面是否互链
- [ ] 数据缺口：可通过网络搜索补充的信息
- [ ] 置信度标注：所有页面是否有 frontmatter confidence；含有 `[推断]`/`[推测]`/`[待验证]` 的声明是否已标注

## 🎯 置信度标注系统

### 为什么需要

Wiki 中每条知识的可信度差异巨大：
- "Anthropic 发表了 Agent 工具设计指南" — 原文白纸黑字
- "Agent 系统的可靠性是当前最大瓶颈" — 多源一致但无人明确定义
- "GPT-5 将在 Q3 发布" — 纯粹的推测

没有置信度标注，查询时无法判断信息的可靠性。

### Frontmatter 置信度

每篇 Wiki 页面在 frontmatter 中标注整体置信度：

| 等级 | 含义 | 适用场景 |
|------|------|----------|
| **high** | 全文主要观点有直接来源支持 | 论文精读、官方文档、作者原文 |
| **medium** | 核心观点可靠，部分推断或未经证实的声明 | 博客综述、行业分析、二手报道 |
| **low** | 大量推断、推测、或来源不可靠 | 观点文章、未经验证的工具测评 |
| **mixed** | 文章内置信度差异大 | 混合了硬数据和观点的分析 |

### 行内标注

在正文中，对特定声明用行内标记注明来源类型：

| 标记 | 含义 | 示例 |
|------|------|------|
| `[原文]` | 来源中的直接引用或转述 | "上下文长度每增加 4x，检索准确率下降 ~5% `[原文]`" |
| `[推断]` | 从多源信息综合推断，非任何单一来源直接陈述 | "Agent 生态正在从工具层向 Harness 层收敛 `[推断]`" |
| `[推测]` | 基于趋势的外推或 LLM 自身的知识补充 | "2027 年之前多数企业 Agent 将采用托管模式 `[推测]`" |
| `[待验证]` | 看起来重要但缺乏来源，需要后续查证 | "OpenAI 内部使用自研 RLHF pipeline `[待验证]`" |

**默认规则：** 如果没有标注，读者应假定为 `[原文]`（来自 sources 中列出的文件）。

### 何时标注

- **必须标注 `[推断]`：** 任何原文没直接说、但通过多源对比得出的结论
- **必须标注 `[推测]`：** 任何涉及未来预测、未公开信息、或 LLM 背景知识的声明
- **必须标注 `[待验证]`：** 写 wiki 时觉得重要但找不到来源的关键事实
- **可选标注 `[原文]`：** 默认就是原文，标注是为了强调"这确实是作者说的"

### Hub 页面特殊规则

Hub 页面（汇总页）通常为 **mixed** 置信度，因为它们综合了多个源：
- 来自细节页的直接引用 → `[原文]`
- 跨细节页的综合 → `[推断]`
- 趋势预测 → `[推测]`

## ⚠️ 约定

1. **raw/ 不可变** — LLM 只读，永远不修改
2. **wiki/ LLM 全权拥有** — 用户偶尔手动调整，但默认 LLM 维护
3. **交叉引用用 `[[]]`** — Obsidian 兼容格式
4. **每个操作都记录到 log.md** — 可追溯
5. **摘要要简洁有力** — 一句话说明页面核心价值
6. **中文为主，英文术语保留原文** — 如 "上下文工程 (Context Engineering)"
7. **来源必须标注** — 每个 Wiki 页面都要标明来自哪些 raw 文件
