# 评估方法

> Agent 评估比传统 NLP 评估复杂得多——涉及交互、环境、工具等多维因素。基础设施配置本身可能比模型差异更大。

## 核心挑战

- **非确定性**：采样和工具失败导致结果波动
- **长时信用分配**：多步任务中难以归因成功/失败
- **环境变异性**：工具、基础设施、沙箱配置都影响分数
- **可复现性**：缺少标准化协议和完整 trace
- **隐藏成本**：重试次数、上下文增长等不易度量

## 关键发现

### 基础设施噪声

基础设施配置可导致 **6 个百分点**的分数波动——超过顶尖模型间的真实差距：
- **1x → 3x**：修复基础设施可靠性（瞬时资源峰值）
- **3x → uncapped**：改变可解性（额外资源让 Agent 尝试更重方案）
- 启示：**资源限制改变了评估的测量内容**，严格限制奖励高效策略，宽松限制奖励能利用资源的策略

→ 详见 [[基础设施噪声]]

### 评估维度与方法

| 维度 | 方法 |
|------|------|
| 任务完成度 | 静态准确率、动态交互评估 |
| 效率 | 步骤数、时间、token 消耗 |
| 鲁棒性 | 异常情况、边缘案例表现 |
| 可靠性 | 一致性和可复现性 |

主流基准：WebArena、ToolBench、SWE-bench、GAIA、ALFWorld

→ 详见 [[Evaluation-Survey]]

### 工具评估

- 用真实场景生成评估任务（非简单沙箱）
- 让 Agent 自身参与评估和优化（LLM-as-judge）
- Claude 优化后的工具准确率显著超过人工编写版本
