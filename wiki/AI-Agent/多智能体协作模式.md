# 多智能体协作模式

> 翻译自 Cara Phillips 的多智能体协作模式指南。核心建议：从最简单的、能跑通的模式开始，观察瓶颈，再逐步升级。

## 五种主流模式

### 模式一：生成-验证者 (Generator-Verifier)

- **机制**：生成者给初稿，验证者评估，循环修改直到通过或达到最大次数
- **适用场景**：输出质量至关重要、有明确评估标准（代码生成+测试、事实核查、合规审查）
- **局限**：
  - 验证者审核标准不细则变成橡皮图章
  - 创意评估难度等同于创意本身
  - 容易死循环

### 模式二：调度-子智能体 (Orchestrator-Subagent)

- **机制**：调度者规划 + 分发 + 汇总，子智能体处理具体任务
- **案例**：Claude Code 使用此模式——主智能体写代码改文件，需要搜索时派给后台子智能体
- **适用场景**：任务拆解明确、子任务互不依赖（代码审查系统）
- **局限**：调度者成为信息瓶颈；不做并行则享受不到速度优势

### 模式三：智能体团队 (Agent Teams)

- **机制**：协调者创建多个独立进程的团队成员，共享任务队列抢单
- **与调度模式区别**：成员是长期存在的，不断积累领域知识
- **适用场景**：子任务独立、需要跨越多步长时间处理（代码库迁移）
- **局限**："独立"是生命线也是软肋；进度管理困难；公共资源竞争

### 模式四：消息总线 (Message Bus)

- **机制**：发布/订阅模式，路由器精准推送消息
- **适用场景**：事件驱动的流水线、智能体团队可能继续扩编（安全运营系统）
- **局限**：排查问题困难；路由器分发错误导致"静默崩溃"

### 模式五：共享状态 (Shared State)

- **机制**：所有智能体共同面对持久化存储区，无中心指挥官
- **适用场景**：需要高度协作、互相参考发现（跨领域综合研究）
- **局限**：容易重复造轮子；最致命是"反应式死循环"（两个智能体无限套娃聊天）
- **关键**：必须设定终止条件——固定时间预算、收敛阈值、或裁判智能体

## 如何选择

| 对比维度 | 选择建议 |
|----------|----------|
| 调度-子智能体 vs 智能体团队 | 子任务短平快选前者，需要长期积累上下文选后者 |
| 调度-子智能体 vs 消息总线 | 工作流可预测选前者，事件驱动选后者 |

**核心原则**：围绕"上下文"来拆解工作（Context-centric decomposition）
