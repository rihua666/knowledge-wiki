# 你不知道的 Agent：原理、架构与工程实践

> Tw93 (@HiTw93) 系统梳理 Agent 架构里最影响工程效果的内容，涵盖控制流、上下文工程、工具设计、记忆、多 Agent 组织、评测、追踪和安全。
> 来源: [X Article](https://x.com/HiTw93/article/2034627967926825175) | 2026-03-19

## Agent 范式：五种控制模式

大多数 AI 系统拆开看都是这五种模式的组合：

### 1. Prompt Chaining（提示链）
任务拆成顺序步骤，每步 LLM 处理上一步的输出，中间可加代码检查点。
- 适合：生成后翻译、先写大纲再写正文等线性流程

### 2. Routing（路由）
对输入分类，定向到对应的专用处理流程。
- 适合：简单问题走轻量模型，复杂问题走强模型；不同类型走不同逻辑

### 3. Parallelization（并行化）
两种变体：分段法（拆成独立子任务并发）和投票法（同一任务跑多次取共识）。
- 适合：高风险决策、多视角场景

### 4. Orchestrator-Workers（编排器-工作者）
中央 LLM 动态分解任务，委派给工作者 LLM，综合结果。
- 适合：编码、研究等多步任务（nanobot spawn、learn-claude-code 子 Agent）

### 5. Evaluator-Optimizer（评估器-优化器）
生成器产出，评估器给反馈，循环直到达标。
- 适合：翻译、创意写作等质量标准难用代码定义的任务

### Workflow vs Agent

**核心区分**：执行路径由代码预先写死的是 Workflow，由 LLM 动态决定下一步的是 Agent。控制权在谁手里决定了本质区别。

> 很多标着 Agent 的产品，深入看其实更接近 Workflow。两者并无高下之分，关键是给任务找到更适合的解决方案。

## 核心观点

### Harness 比模型更关键
Harness = 验收基线 + 执行边界 + 反馈信号 + 回退手段。更贵的模型提升很多时候不如 Harness 和验证测试质量对成功率的贡献大。

**任务矩阵**（任务清晰度 × 验证自动化程度）：
- 右上角（目标明确 + 可自动验证）= 最适合 Agent
- 要做的就是把任务推进右上角，让对错有机器可执行的判断标准

### 上下文工程
- **分层管理**：常驻层（身份/约束）→ 按需加载（Skills）→ 运行时注入 → 记忆层 → 系统层
- **Prompt Caching**：稳定的大系统提示比频繁变动的小提示实际成本更低（前缀命中率）
- **Skills 延迟加载**：描述符是路由条件不是功能介绍；必须加反例（准确率从 53%→85%）
- **压缩保留优先级**：架构决策 > 已修改文件 > 验证状态 > 未解决 TODO > 工具输出

### 工具设计演进
- 第一代：API 封装（粒度过细）
- 第二代：ACI（面向 Agent 目标而非底层 API）
- 第三代：Advanced Tool Use（动态发现 + 代码编排 + 示例驱动）

**调试 Agent 时应先检查工具定义**，多数工具选择错误出在描述不准确。

### 记忆系统四层
1. **工作记忆**：上下文窗口，主动管理
2. **程序性记忆**：Skills，按需加载
3. **情景记忆**：JSONL 会话历史，持久化可检索
4. **语义记忆**：MEMORY.md，Agent 主动维护

### 多 Agent 协作
- **指挥者模式**：同步协作（人与单 Agent 紧密互动）
- **统筹者模式**：异步委派（人只在起点和终点出现）
- **关键顺序**：协议先定 → 隔离先做 → 再谈协作和并行
- **子 Agent 只回传摘要**，探索细节留在自己的上下文里
- **幻觉互相放大**：需要交叉验证打断错误传播链

### 评测
- **Pass@k**（能力边界）vs **Pass^k**（回归测试）不能混用
- **先修评测再改 Agent**：评测系统出问题时改 Agent 方向可能是错的
- **基础设施噪声**：资源不足导致进程被杀 ≠ 模型失败

### 长任务稳定运行
- **状态外化**：进度放文件不放进上下文（JSON > Markdown）
- **Initializer Agent + Coding Agent**：先拆任务再循环执行
- **后台 I/O**：慢速操作放后台线程，通知队列注入结果

### 常见反模式
1. 系统提示当知识库
2. 工具数量失控
3. 验证闭环缺失
4. 多 Agent 无边界
5. 记忆不整合
6. 没有评测
7. 过早引入多 Agent
8. 约束靠期望不靠机制
