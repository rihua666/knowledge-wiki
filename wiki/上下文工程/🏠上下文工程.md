# 上下文工程 (Context Engineering)

> 上下文工程是 prompt 工程的自然延伸——不是写好一个提示词，而是策略性地策展和管理 LLM 推理过程中的全部 token 信息。

## 核心问题

LLM 的上下文是一个**有限资源**：
- **Context Rot（上下文腐烂）**：context window 越大，模型准确回忆信息的能力越差
- **注意力预算**：基于 transformer 的 n² 注意力关系，每个新 token 都消耗预算
- **边际收益递减**：必须精心策展高信号 token

> 指导原则：**找到最小的高信号 token 集合，最大化期望结果的概率。**

## 四大技术方向

### 1. 静态上下文优化

精心设计始终包含在上下文中的内容：
- **System Prompt**：找到"正确高度"——不过度硬编码，也不过于模糊
- **工具设计**：自包含、容错、用途明确，避免功能重叠
- **Few-shot 示例**：精选多样化示例，而非罗列边界情况

→ 详见 [[有效上下文工程]]

### 2. 动态上下文发现

预先提供更少的上下文，让 Agent 按需自主提取：
- **Just-in-Time 策略**：维护轻量标识符，运行时动态加载
- **渐进式披露**：Agent 逐层发现相关上下文
- **工具响应转文件**：大型 JSON 写入文件，Agent 用 tail 按需读取

→ 详见 [[动态上下文发现]]

### 3. 工程优化实践

生产环境中的关键指标和技巧：
- **KV-cache 命中率**是最重要的单一指标（Claude Sonnet 缓存 vs 未命中成本差 10 倍）
- **掩蔽而非移除工具**：避免 KV 缓存失效和模型困惑
- **文件系统作为外部记忆**：大小不限、天然持久化、可恢复压缩
- **复述操控注意力**：todo.md 等结构化笔记强制 Agent 聚焦

→ 详见 [[Manus-经验]]

### 4. 架构级扩展

当单上下文窗口不够时的架构解决方案：
- **Compaction（压缩）**：接近 context 上限时摘要压缩
- **结构化笔记**：Agent 定期写入持久化笔记，跨 session 保持记忆
- **子代理架构**：主 Agent 协调，子 Agent 专注执行，只返回压缩摘要
- **并行处理**：部署 n 个子代理处理 n 个项目（彻底避免幻觉阈值）

→ 详见 [[Wide-Research]]
