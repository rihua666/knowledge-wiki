# AI代理的上下文工程：构建Manus的经验教训

> Manus 团队重建了 4 次 Agent 框架，分享了围绕 KV 缓存、动态动作空间、文件系统记忆和注意力操控的实战经验。

## 核心内容

### 核心选择：上下文工程而非端到端训练

- 上下文工程允许几小时而非几周内交付改进
- 产品与底层模型正交：模型进步是潮水，Manus 是船
- 称优化过程为"随机研究生下降 (SGD)"
- Manus 的上下文工程实践是 [[有效上下文工程]] 中 Anthropic 理论框架的生产级验证

### 1. 围绕 KV 缓存进行设计

**KV-cache 命中率是生产阶段最重要的单一指标。** 这与 [[有效上下文工程]] 中关于上下文是有限资源的论述直接相关。

- Manus 平均输入:输出 token 比约 100:1
- Claude Sonnet 缓存命中 vs 未命中成本差 10 倍（$0.30 vs $3.00/百万 token）

关键实践：
- 保持提示前缀稳定（不要在系统提示开头放时间戳）
- 上下文只追加，不修改之前的操作
- 明确标记缓存断点

### 动态动作空间管理

工具数量爆炸（MCP 加剧），但动态增删工具会导致 KV 缓存失效。这与 [[Writing-Tools]] 中"避免臃肿工具集"的建议互补。
1. KV 缓存失效（工具定义在上下文前部）
2. 模型困惑（先前引用的工具消失导致幻觉）

解决方案：**上下文感知的状态机**，通过 logits 掩蔽约束动作选择：
- 工具命名一致前缀（browser_*、shell_*）
- 响应预填充约束（auto/required/specified 三种模式）

### 3. 文件系统作为上下文

128K 上下文仍不够，截断压缩导致信息丢失。Manus 的方案：

- **文件系统 = 终极上下文**：大小不限、天然持久化、Agent 可直接操作
- 这与 [[A-Mem]] 的结构化笔记理念不同——A-Mem 用 LLM 生成丰富的语义笔记，Manus 更侧重工程实用性
- 压缩策略设计为可恢复：保留 URL → 网页内容可移除；保留路径 → 文档内容可省略
- 模型学会按需写入和读取文件 = 结构化外部记忆

### 4. 通过复述操控注意力

Manus 创建 todo.md 并逐步更新——这不是可爱行为，而是**操控注意力的刻意机制**：
- 创建结构化待办清单 = 强制 Agent 复述和结构化思维
- 复述迫使 Agent 将注意力集中在任务的关键方面
- 类比 CoT：将内部推理外部化

## 关键要点

- 上下文工程是实验科学，需要反复迭代
- KV 缓存是生产环境的首要优化目标
- 文件系统作为外部记忆是解决上下文限制的根本方案
- 注意力操控（复述）是提升复杂任务表现的关键技巧
