# 有效上下文工程 (Effective Context Engineering)

> 上下文工程是 prompt 工程的自然延伸，核心是在有限注意力预算下，为 AI Agent 策略性地策展和管理上下文信息。

## 核心内容

### 上下文工程 vs Prompt 工程

- **Prompt 工程**：如何写好系统提示词（静态、一次性）
- **上下文工程**：管理整个推理过程中的所有 token（系统指令、工具、MCP、外部数据、消息历史等），是**迭代式**的策展过程

### 上下文是有限资源

- **Context Rot（上下文腐烂）**：context window 越大，模型准确回忆信息的能力越差
- **注意力预算**：基于 transformer 的 n² 注意力关系，每个新 token 都在消耗预算
- **边际收益递减**：上下文必须被视为有限资源，需精心策展高信号 token

### 有效上下文的组成要素

1. **System Prompt**：找到"正确高度"——既不过度硬编码逻辑，也不过于模糊
   - 用 XML 标签或 Markdown 标题组织不同部分
   - 追求**最小但完整**的信息集
   - 先用最强模型测试最小 prompt，再根据失败模式添加指令

2. **工具设计**：工具定义 Agent 与信息/动作空间的契约
   - 自包含、容错、用途明确
   - 避免臃肿的工具集（功能重叠会导致 Agent 困惑）
   - 返回 token 高效的信息

3. **Few-shot 示例**：精选多样化、规范化的示例，而非罗列所有边界情况

### 上下文检索与自主搜索

- **Just-in-Time 策略**：Agent 维护轻量标识符（文件路径、查询、链接），运行时动态加载
- **渐进式披露 (Progressive Disclosure)**：Agent 逐层发现相关上下文
- **混合策略**：部分预计算 + 部分运行时探索（如 Claude Code：CLAUDE.md 预加载 + glob/grep 实时探索）

### 长时任务技术

| 技术 | 适用场景 | 核心思想 |
|------|----------|----------|
| **Compaction（压缩）** | 需要大量来回对话的任务 | 接近 context 上限时摘要压缩，丢弃冗余工具输出 |
| **Structured Note-taking（结构化笔记）** | 有明确里程碑的迭代开发 | Agent 定期写入持久化笔记（如 NOTES.md），跨 session 保持记忆 |
| **Sub-agent Architecture（子代理架构）** | 复杂研究和并行探索 | 主 Agent 协调，子 Agent 专注执行，只返回压缩摘要 |

## 关键要点

- 上下文工程的核心原则：**找到最小的高信号 token 集合，最大化期望结果的概率**
- 工具设计要像好的代码库——自包含、容错、清晰
- 随着模型能力提升，所需的预设工程减少，Agent 自主性增加
- 即使 context window 变大，上下文污染和信息相关性问题仍然存在
- "做最简单有效的事"——当前最佳实践建议
